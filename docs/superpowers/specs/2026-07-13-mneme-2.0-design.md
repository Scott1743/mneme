# Mneme 2.0 — LLM Wiki 重构设计

> 立项：2026-07-13
> 版本：2.0.0（设计文档）
> 状态：brainstorming 已批准；等待 Superpowers writing-plans 与 gstack plan-eng-review 落地。

---

## 1. 背景与目标

Mneme 1.1.0 已经把交付收敛为单一 skill zip，并把 OKF v0.1 作为 wiki 本体的格式契约。但当前实现把产品叙事扭曲成了"CLI + 可选 L2 向量检索"，并且：

- 用户表面暴露了 4 个 CLI 子命令（`init / reindex / search / lint`），而 `dream` 被明确冻结——与"agent 维护的有标签 Markdown wiki"定位脱节。
- `SKILL.md` 中 `Scenario: query … Naive RAG: embed → KNN → top-k → read pages → synthesize`，反复把叙事推向 RAG。
- 1.1.0 的 CHANGELOG 与实际行为不一致（声称 lazy install + `ensure_index_deps()`，但 `skills/mneme/scripts/mneme/` 不存在 `lazy_index.py`）。
- `.github/workflows/ci.yml` 仍跑 `python -m build --wheel` 与 `pip install -e '.[dev,validate,index,toml10]'`，与 "no wheel, zip-only delivery" 矛盾。
- `skills/mneme/scripts/mneme.egg-info/` 仍在 working tree，违反 v1.1.0 的交付模型承诺。

2.0 的目标是回到 Karpathy *LLM Wiki* 与 Google OKF v0.1 的根本：

- **知识"编译一次"，不是每次提问都做 RAG**。
- **唯一真相源是一棵 OKF 合规、本地、可被 git 管理的 Markdown 树**。
- **agent 是唯一的维护者与查询员**；用户只与 agent 对话。

## 2. 产品边界

### 2.1 用户视角只看见 `dream` 与 `search`

- **`dream`：写侧总入口**
  - 带资料（文件路径、文本、URL、思想）：agent 摄入并蒸馏成 OKF 概念页，打标签、互链，更新 `index.md` 与 `log.md`。
  - 不带资料：agent 对 wiki 做安全体检，输出"应当如何修改"的预览；只有用户明确指示的修改才落盘。
- **`search`：读侧总入口**
  - agent 先沿 `index.md`、frontmatter `tags`、Markdown 链接、文本匹配定位，再读取完整页面综合回答。
  - 规模上来后可调用本地 SQLite / FTS / vec / BGE 索引导航，但索引命中的 chunk 永远不能直接当答案。
- `init / lint / reindex` 是 agent 内部按目录/索引/查询状态**语义执行**的脚本，对用户不暴露为产品能力。

### 2.2 四层架构

| 层 | 角色 | 拥有者 | 落地位置 |
|---|---|---|---|
| Raw sources | 不可变原始资料 | 用户只读 | `wiki/sources/` 或外部引用 |
| OKF Wiki | 唯一真相源 | LLM agent 全权维护 | `wiki/` |
| Agent Skill | 行为规约 | 人 + agent 共写 | `~/.claude/skills/mneme/SKILL.md` + `references/` |
| Disposable accelerator | 可删除的导航缓存 | 不拥有事实 | `wiki/.mneme/index.db` |

### 2.3 标签（`tags`）规则

- OKF v0.1 的 `type` 是协议级 MUST；**不为 OKF 新增 MUST**。
- Mneme **自己写入**的概念页要求至少一个 `tags` 值——这是 Mneme 写作纪律，不是上游协议。
- 消费**外部 OKF bundle** 时，缺 tags 只告警、不拒绝、不改写。
- `type` 描述文档角色；`tags` 描述主题归类；Markdown 链接表达具体关系。三者语义不重叠。
- 不建立全局标签 taxonomy；词汇由每座 wiki 在 `index.md` 中渐进成长。
- 不强制生成 `tags/<tag>.md` 镜像聚合页——这些会增加维护成本并容易腐化；agent 若需要阅读地图，可写普通 OKF `Topic` 页面，不强制。

## 3. 组件与数据流

### 3.1 Skill 与脚本分工

- `SKILL.md`：只写意图判断与 agent 工作流（dream / search），不出现 Python 调用细节。
- `scripts/mneme/` Python 脚本：只承担确定性操作——
  - `init`：创建最小 OKF bundle（`index.md`、`log.md`、`.mneme/`）。
  - `lint`：检查 OKF 硬规则 + Mneme 写作纪律（tags 健康、断链、孤儿、保留文件）。
  - `reindex`：从 Markdown 全量重建可删除索引。
  - `search`：返回候选路径与 distance，不生成答案。
- 没有 Python `dream`/`ingest`/`query` 模块——这些动作需要语言理解与价值判断，由 host agent 完成。

CLI 仍然存在，便于测试和 agent 调用；但 README、`introduction/`、`SKILL.md` 的用户入口只讲 `dream / search`。

### 3.2 `dream` 数据流

1. 解析 bundle；不存在时 agent 语义调用 `init`。
2. **带资料**：
   1. 保存或引用 source（不可变）；
   2. 读取相关既有页，判断新增 / 合并 / 更新；
   3. 写概念页（含 `type / title / description / tags / timestamp / resource` frontmatter）；
   4. 用 Markdown 链接表达关系；
   5. 更新 `index.md`，向 `log.md` 顶部追加一致前缀条目；
   6. 调用 `lint`；成功后再按需 `reindex`。
3. **无资料**：
   1. 只读扫描并生成 dream 预览（重复候选、孤儿、标签漂移、缺链、可能过时）；
   2. 默认禁止自动 commit、push、建分支、`git add -A`；
   3. 用户确认后，agent 做定向修改，再 `lint / reindex`。
4. 任何写操作以 Git diff 可审查为目标；禁止以"向量相似度"自动删除、合并或归档事实页。

### 3.3 `search` 数据流

1. agent 先读 `index.md`，按 tags / 标题 / 链接 / `grep` 渐进展开；
2. 必要时调用内部本地索引（依次降级）：
   - **L0**（默认）：`index.md` + tags + 链接 + grep —— 零依赖、永远可用。
   - **L1**：`sqlite3` 内建 + FTS5（默认开启；首次 `reindex` 创建 `pages_fts`）—— 标题、描述、tags、正文、链接的全文检索。
   - **L2**：`sqlite-vec` + FastEmbed（默认 `BAAI/bge-small-zh-v1.5`）—— 语义召回，仅用于大 wiki。
3. L2 仅返回候选路径、heading、原始 distance；agent 必须 `Read` 完整 Markdown 页面；
4. 答案引用 bundle-relative 路径；找不到时明确说"wiki 中没有"，禁止模型常识伪装；
5. 高质量回答可由 agent 提议"用 `dream` 回填"，但 `search` 自身只读。

### 3.4 "内置 SQLite / vec / BGE" 的实际含义

- `sqlite3` 是 Python 标准库——`index.db` 默认 L1 底座，不增加依赖。
- skill zip **不打包**平台相关 sqlite-vec 二进制，也不打包 ~90 MB 的 BGE ONNX。
- 首次真正需要 L2 时，由 agent 透明说明：`Bash: python3 ~/.claude/skills/mneme/scripts/mneme.py bootstrap-vec`（或等价语义调用），依赖装到当前 venv 或 `--user`，模型缓存到 `~/.cache/mneme/models`。
- 离线、用户拒绝、平台不支持、下载失败时，自动降级到 L1 + 链接——**wiki 本身始终可用**。
- 用户拒绝 L2 不影响 1.1.0 已有的 OKF core 工作流；2.0 也走同一契约。

## 4. 安全与错误处理

- 沿用 OKF v0.1 §9 容错消费契约：缺失可选字段、未知 `type`、未知额外 frontmatter 键、断链、缺 `index.md` 都不能拒绝整个 bundle；坏文件不影响其他文件。
- Mneme 写入页缺 `tags` 视为 Mneme 质量错误（lint WARN/ERROR，可配置），读取外部 OKF 时只 WARN。
- Source 默认不可变；`dream` 只能改写 wiki 层。
- Dream 无资料模式必须先预览；禁止按未经人工校准的向量阈值自动合并 / 删除 / 归档。
- 不自动 `git add -A`、commit、push；不自动建 `dream/<date>` 分支。
- 索引损坏直接删除并全量重建，永远不尝试就地修复成新的事实来源。
- sqlite-vec / FastEmbed / 模型不可用 → 降级到 L1 + 链接 + grep，wiki 仍可用。
- 所有外部发布动作只在完整验证通过后执行；用户已授权 GitHub 与 skill.sh 发布。

## 5. 测试门禁

### 5.1 OKF 与标签
- OKF 三条 MUST + 保留文件 + 容错消费契约。
- Mneme 写入页至少 1 个 `tags`；外部无 tag 文件仍可消费。
- 未知 frontmatter 键完整保留（白盒 frontmatter dict）。

### 5.2 Dream
- 空 bundle 自动 `init`。
- source → 概念页 → tags/links → `index.md` / `log.md` → `lint` → `reindex` 全链路。
- 无资料 dream 默认 report-only / preview-only：bundle 字节级不变。
- 不调用 `git add -A`、不自动 commit/push、不按向量距离自动改事实页。
- `lint` 失败不得宣称成功。

### 5.3 Search
- `index/tags/links/grep` → SQLite FTS → vec/BGE 的降级顺序必须可重现。
- L2 缺失、离线、索引损坏均有可用回退。
- 搜索只返回候选与 distance；最终答案引用完整页面路径。
- 中文、tags、链接和标题召回测试；保留 `fake_embedder` 的可重现单测。

### 5.4 Dream 的 safety TDD（同时也是 0.2.1 冻结条件的满足证据）
- `test_dream_no_resource_writes_in_dry_run`：dream dry-run 后 bundle mtime + 字节级哈希不变。
- `test_dream_does_not_shell_git_add_a`：监控 `subprocess.run` / `os.execvp` 不可被梦呼到 `git add -A`。
- `test_dream_non_git_repo_graceful_degrades`：非 git 仓库不报错也不自动 commit。
- `test_dream_similarity_reports_raw_distance_only`：报告只列 raw distance，绝不预测 `similarity ≥ 0.92`。
- 反向：`test_cli_dream_requires_apply_flag_to_write`：dream 必须用 `--apply` 显式开闸才允许写。

### 5.5 交付与 CI
- `__version__` 与 `SKILL.md` frontmatter `version` 与 zip 文件名三重一致，目标 **`2.0.0`**。
- zip 内容字节级与 `skills/mneme/` 对应，排除缓存、`*.egg-info`、`.pyc`、`.pyo`、`.DS_Store`、Pages 工程文件。
- CI 彻底删除 `python -m build --wheel` 与 `pip install dist/mneme-*.whl`；只测 zip + skill 入口。
- README、introduction、SKILL 不再把四个 CLI 或"naive RAG"当产品主叙事。

## 7. 发布顺序

1. 全部 §5 测试在 default + `make test-full` 下绿；`make zip` 只生成 `dist/mneme-2.0.0.zip`。
2. `git add -A && git commit` on `refactor/mneme-2.0`；annotated tag `v2.0.0`。
3. `git push origin refactor/mneme-2.0 --follow-tags`；开 PR → main 或直接合并（用户偏好后定）。
4. GitHub Release 上传 zip；记录 release notes 与 check-sum。
5. 验证 `introduction/` Pages 重新部署并通过 §6 中链接测试。
6. 按 README.md 路径触发或检查 skill.sh 索引（无法直接操作 skill.sh，只能给最终链接 + 监控）。
7. 在 CHANGELOG 与 release notes 中明示 "dream 默认 preview-only；`--apply` 才能落盘"。

## 8. 非目标

- 不实现 MCP server。
- 不引入固定 `type` 或 `tags` taxonomy。
- 不实现云 / 远端索引。
- 不引入依赖工具替代 git diff 作为写操作的审计来源。
- 不为 L2 引入非 sqlite-vec + FastEmbed + BGE 的组合。

---

## 9. 风险与后续

- **bootstrap-vec 的可重现性**：不同平台 sqlite-vec wheel 安装常失败；如果实验下来 bootstrap-vec 失败率高，就推迟 L2，把它列入 v2.1。
- **dream preview 的可用性**：报告不能太啰嗦，否则用户会跳过；统一为一份 `dream-report-<date>.md`。
- **GitHub Pages 与 skill.sh 索引延迟**：若 skill.sh 没有自动 resync，需手动 Resync；这是 1.1.0 已知流程。
- **兼容 1.1.0 bundle**：1.1.0 仓库工作树残留 `mneme.egg-info/`、`tests/__pycache__/`、`dist/mneme-1.1.0.zip` 等，必须清理后再提交。
