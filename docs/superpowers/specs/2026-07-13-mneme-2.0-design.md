## 1. Thesis (the one paragraph that must never drift)

Mneme is an **agent-maintained, locally-grown, OKF-conformant, tagged Markdown knowledge wiki**. The user interacts with it **only through an agent** and only with two intents: `dream` (write side — turn inputs into compounding knowledge) and `search` (read side — locate authoritative pages and synthesize from them). The Markdown, its YAML frontmatter, its links, and `index.md` / `log.md` are the **single source of truth**. LLM Wiki is **not** RAG: knowledge is compiled once into the wiki and then *grows*, instead of being re-discovered every question. Retrieval over an SQLite index (and optionally over sqlite-vec + a BGE embedding model) is a **disposable navigation accelerator** that is allowed to disappear without breaking the wiki.

This is the lens every other decision in this document must satisfy.

---

## 2. Product surface

Mneme 2.0 ships exactly **two user-facing intents**, both surfaced only through the host agent:

| Intent  | Direction | What the user means                                                   | What the agent must never do                                          |
|---------|-----------|-----------------------------------------------------------------------|-----------------------------------------------------------------------|
| `dream` | write     | "Compile this into my wiki / tidy the wiki / audit the wiki"          | Hide destructive edits, commit to git, push, treat vector distance as proof |
| `search`| read      | "Find what my wiki says about X"                                      | Answer from outside the wiki, splice chunks directly into an answer   |

Anything that is currently in the CLI / skill / docs as a user-facing verb other than `dream` and `search` (notably `init`, `lint`, `reindex`) **moves out of the product surface** and into the deterministic internal API that the agent calls while serving `dream` or `search`.

## 3. Four-layer architecture

```
┌──────────────────────────────────────────────────────────────────────┐
│ L1 — Raw sources (immutable)                                          │
│   • The host filesystem that the user pointed at                     │
│   • Agent reads; agent never writes                                  │
├──────────────────────────────────────────────────────────────────────┤
│ L2 — OKF wiki (single source of truth)                                │
│   • Ordinary Markdown + YAML frontmatter                             │
│   • Filesystem links carry relationships; surrounding prose          │
│     expresses semantics (per OKF §10)                                │
│   • `index.md` (per-bundle) + `log.md` (newest-first ISO dates)      │
├──────────────────────────────────────────────────────────────────────┤
│ L3 — Agent skill (this delivery)                                      │
│   • `SKILL.md` defines the `dream` and `search` workflows            │
│   • Internal scripts implement deterministic operations:             │
│     `init`, `lint`, `reindex`, `search-index`                        │
│   • The skill teaches the agent the Mneme maintenance discipline      │
├──────────────────────────────────────────────────────────────────────┤
│ L4 — Disposable accelerator (`<bundle>/.mneme/index.db`, gitignored)  │
│   • SQLite (stdlib) always available; FTS5 when the bundle has it   │
│   • sqlite-vec + FastEmbed (BGE) is opt-in                           │
│   • Rebuilt from scratch by `reindex`; never edited in place         │
│   • Loss of this layer must NOT corrupt the wiki                     │
└──────────────────────────────────────────────────────────────────────┘
```

L4 is allowed to vanish. L2 must not.

## 4. What Mneme 2.0 *does* to OKF (and what it explicitly does not)

OKF v0.1 §9 is unchanged. Mneme 2.0:

- **Does** require `type` on every Mneme-authored non-reserved `.md` (mirrors OKF MUST).
- **Does** require at least one `tags` value on every Mneme-authored non-reserved `.md`. This is a **Mneme production rule**, not an OKF protocol change: OKF §4.1 leaves `tags` recommended; we narrow it to required for *our own writes*. When consuming an external bundle, missing `tags` is **a warning, not a refusal** — the OKF lenient-consumption contract (§9) is preserved.
- **Does not** introduce a fixed `type` taxonomy, a fixed `tags` vocabulary, or a parallel OKF layer.
- **Does not** generate mirror `tags/<tag>.md` aggregation pages by default. When the agent wants a curated reading map, it writes a normal OKF `Topic` page (its `type` value) and links to the relevant pages.

The `type` field documents the role of a page (e.g. `Concept`, `Reference`, `Summary`, `Source`, `Topic`); `tags` documents topical affinity; Markdown links document the relationship. They are three different axes and must not collapse into one.

## 5. The `dream` workflow

### 5.1 With input material

1. Resolve the bundle (config → `MNEME_BUNDLE` → walk-up `index.md` → `./wiki`). If missing, the agent calls the internal `init` script.
2. Materialize the input as a **read-only source** (path reference or copy into `sources/`).
3. Read relevant existing pages; decide whether the input is new, a duplicate, or an update.
4. Author the OKF pages:
   - `type`, `title`, `description`, `tags: [at least one]`, `timestamp` (ISO 8601), optional `resource`.
   - Express relationships with Markdown links (per OKF §10); the prose around each link gives its meaning.
5. Edit `index.md` to reflect new pages (no fixed taxonomy; just human-maintained grouping).
6. `prepend` an entry to `log.md` with `## YYYY-MM-DD ingest | <title>` (the format the existing test suite asserts on).
7. Call internal `lint`. If lint fails, the change is **not committed** to the wiki's `log.md` either; the agent reports the failure.
8. Call internal `reindex` only if L4 is present (L4 is allowed to be absent; absence is not an error).
9. The wiki, L1, and L3 remain the source of truth. L4 is regenerated; it never accumulates truth.

### 5.2 Without input material (audit-only)

`dream` without inputs is the curated-tidy mode. It must:

- Default to **report-only / preview-only**. No edits land on disk without explicit user acceptance.
- Be free to suggest:
  - candidate merges of near-duplicate pages (with raw sqlite-vec distance and the embedding model id, **not** a fabricated similarity percentage);
  - candidates for `archive/` (orphans, very old pages with no recent `log.md` reference);
  - missing backlinks, broken links, tag drift;
  - candidate `Topic` reading maps.
- **Never** auto-commit, auto-push, run `git add -A`, or alter facts based on raw embedding distance. The historical v0.2.1 P0 freeze on `dream` (similarity math wrong, `find_orphans` missing, `git add -A` ordering, pseudo-transactional staging, no rollback test) stays in force and is the source of truth for "do not ship these by default".

The agent may downgrade one explicit `--apply` switch into a follow-up implementation plan only after the preview path has a dedicated safety TDD suite (see §10). v2.0 ships preview-only.

### 5.3 Git discipline

- Dream only commits changes it has just authored; it never runs `git add -A`.
- Dream never pushes. It never opens a PR.
- The wiki history is always inspectable through ordinary `git log` / `git diff` / `git blame`.

## 6. The `search` workflow

1. Read `index.md` and follow the OKF links / `tags` to candidate pages.
2. Use the host agent's own `grep` / `glob` over the bundle.
3. If the bundle has L4 (`<bundle>/.mneme/index.db`):
   - Run internal `search-index "<query>" --json` and consume its output as a **navigation** candidate set:
     - page path, nearest heading, raw sqlite-vec distance, embedding model id.
   - L1 path (no sqlite-vec): plain SQLite FTS5 if available, else linear title/tags/scan.
   - L2 path (sqlite-vec + FastEmbed): cosine distance is logged, not normalized, not compared to a hardcoded "0.92"-style threshold for any decision.
4. For every candidate, the agent **reads the full Markdown page** before composing any part of the answer.
5. Citations are bundle-relative paths. "The wiki does not say X" is an acceptable and required answer.
6. `search` is read-only. It never writes back into L2; that is `reindex`'s job. A particularly good answer may be **proposed** for `dream`, but `search` itself never drafts new pages.

### 6.1 Graceful degradation matrix

| L4 state                                | `dream` ingest path          | `search` path                   |
|-----------------------------------------|------------------------------|---------------------------------|
| absent                                   | full functionality           | index + tags + links + grep     |
| FTS5 only (no sqlite-vec)                | full functionality           | FTS5 → page read                |
| sqlite-vec + FastEmbed                   | full functionality           | vec → page read                 |
| corrupt (size mismatch / schema fail)    | full functionality           | `reindex` suggested, falls back |

When L2 is missing or broken, `dream` and `search` must still answer. The wiki must always remain navigable.

## 7. Dependency model

### 7.1 Zero-dep baseline (always)

- Python ≥ 3.10, stdlib only.
- `okflib` (frontmatter parsing, OKF validation, orphan find, lint primitives).
- Internal CLI: `init`, `lint`, `reindex`, `search-index`.

### 7.2 L2 dependencies (opt-in, lazy)

- The skill zip does **not** embed:
  - `sqlite-vec` platform-specific binaries
  - FastEmbed
  - any embedding model weights
- The first time `search-index` (or an `reindex` of the vector portion) is invoked without the optional deps, the skill:
  1. prints a one-line `pip install` instruction,
  2. does **not** auto-install (the user decides),
  3. downgrades silently to the next-faster available path.
- Model cache lives under the user cache (`~/.cache/mneme/models` on Linux, `~/Library/Caches/mneme/models` on macOS) and respects the `MNEME_MODEL_CACHE` env override.

This is exactly the discipline v1.1.0 committed to in `SKILL.md` / `README.md` / `test_zero_dep.py`. The CHANGELOG entry that claimed automatic `pip install` in 1.1.0 was **inaccurate**; 2.0 must not perpetuate that fiction (see §10.5).

## 8. Delivery contract

- **Delivery object:** one zip at `dist/mneme-<version>.zip`, contents byte-identical to `skills/mneme/` (with `__pycache__`, `*.egg-info`, and any other build artifacts excluded).
- **Skill shape on disk:** `~/.claude/skills/mneme/` (zip root prefix `mneme/`, per `tests/test_release_layout.py:43-67`).
- **Distribution channels:**
  - GitHub release zip (the primary artifact).
  - skill.sh via the standard `npx skills add Scott1743/mneme` flow.
- **Versioning:** `2.0.0`. Bumped in lockstep at:
  - `skills/mneme/scripts/mneme/__init__.py::__version__`
  - `skills/mneme/SKILL.md` YAML frontmatter `version`
  - zip filename, asserted via test
  - `CHANGELOG.md`
- **No wheel.** No `[project]` table in `pyproject.toml`. CI does not invoke `python -m build` or `pip install dist/mneme-*.whl`. The historic `wheel-smoke` job is deleted.

## 11. Error and edge-case handling

- `find_orphans` failure → returns empty list + warning; lint proceeds.
- `lint` failure after `dream` → change is rolled back from `index.md` / `log.md`; the agent surfaces the failure verbatim.
- `reindex` failure → L4 stays as it was; L1/L2 unchanged; `search` falls back.
- sqlite-vec load failure at runtime → downgrade to FTS5 or linear scan; never crash the agent loop.
- Corrupt `index.db` → suggest `reindex`; do **not** delete silently.
- Non-git bundle → `dream` works; just skip git hygiene steps; warn once.
- External bundle missing `tags` on existing pages → `dream` only complains when it tries to author a new page there; it never mutates the user's existing pages to add tags.

## 12. Out of scope for 2.0

- v2 converters (Word / PDF / PPT / Excel / image / HTML → md/csv) — deferred.
- MCP server — deferred.
- `dream --apply` automatic merging based on vector thresholds — explicitly out of scope.
- `tags/<tag>.md` mirror pages — explicitly out of scope.
- A fixed `type` taxonomy — explicitly out of scope.
- Remote/managed deployment — explicitly out of scope.
