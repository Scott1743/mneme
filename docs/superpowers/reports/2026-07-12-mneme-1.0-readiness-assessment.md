# Mneme 1.0 Readiness Assessment

- **Date:** 2026-07-12
- **Scope:** repository architecture, OKF correctness, CLI/runtime paths, skill workflows, packaging, tests, retrieval quality, and resource use
- **Baseline:** commit `e6d4b69` (`mneme 0.2.0`) plus the uncommitted generated-dist ignore rule
- **Conclusion:** **not ready for 1.0.0**. The L1/L2 prototype is viable and the real semantic search path runs, but several advertised workflows are incomplete or unsafe, and the validator can certify invalid bundles.

## Executive assessment

Mneme has a sound central idea and a better research foundation than its implementation maturity suggests. The strongest parts are the plain-Markdown source of truth, the explicit OKF tolerance contract, the separation of derived indexing from L1 content, atomic index replacement, and a small fast unit-test suite.

The project is currently a **credible 0.x prototype**, not a release candidate. The gap is not primarily cosmetic. Four contracts are presently false:

1. `validate_okf.py` does not establish OKF YAML conformance.
2. The documented lint and dream workflows are not executable as written.
3. The installed/distributed artifact does not provide the command paths promised by the skill.
4. Retrieval behavior has no relevance benchmark, and dream uses the vector distance incorrectly.

Recommended overall score: **5.5/10**.

| Dimension | Score | Assessment |
|---|---:|---|
| Design rationale | 8/10 | Clear lineage from LLM Wiki to OKF; good boundary thinking |
| Core L1/L2 implementation | 7/10 | Real init/reindex/search works; atomic rebuild is a solid choice |
| OKF correctness | 3/10 | Validator produces false conformance results |
| Workflow completeness | 3/10 | lint is broken; ingest and dream contradict core invariants |
| Retrieval rigor | 4/10 | Real model works, but no benchmark and several mathematical/chunking gaps |
| Packaging/release | 3/10 | Mixed skill/package model; wheel and dist are incomplete in different ways |
| Resource discipline | 6/10 | L1 is tiny; optional L2 is reasonable but currently wasteful in avoidable places |
| Maintainability | 5/10 | Good module boundaries, offset by duplicated/stale instructions and prose-only behavior |

## Method

The assessment used four evidence sources:

1. Local authority: `.research/upstream/OKF-SPEC.md`, `.research/constraints.md`, and `.research/design-rationale.md`.
2. Design history: all superpowers specs/plans through v2.2.
3. Executable checks: unit tests, invalid-bundle counterexamples, packaging inspection, a fresh production-model reindex/search, and resource measurement.
4. External verification:
   - [Google OKF announcement](https://cloud.google.com/blog/products/data-analytics/how-the-open-knowledge-format-can-improve-data-sharing/)
   - [sqlite-vec repository](https://github.com/asg017/sqlite-vec)
   - [FastEmbed getting started](https://qdrant.github.io/fastembed/Getting%20Started/)
   - [BGE small Chinese model card](https://huggingface.co/BAAI/bge-small-zh-v1.5)
   - [PyPA pyproject guide](https://packaging.python.org/en/latest/guides/writing-pyproject-toml/)

External material was used only to verify dependencies and engineering behavior. The checked-in OKF SPEC remains the conformance authority.

## What works today

### Core execution

- `pytest`: **41 passed** in approximately **0.5 seconds wall time**.
- A fresh temporary bundle successfully completed `init -> production reindex -> search`.
- First production reindex downloaded and initialized `BAAI/bge-small-zh-v1.5`, then indexed 2 concepts/2 chunks in **11.13s**.
- Cached reindex completed in **0.33s**.
- Warm CLI searches completed in **0.31s** and ranked the expected sample concept first.
- Search JSON is stable and paths point back to Markdown source pages.
- Reindex builds a temporary database and atomically replaces the live index, preserving the old index on failure.
- Deleted/moved concepts and `archive/` are excluded after rebuild.
- Index metadata records schema, model, dimension, counts, and sync time.

### Architectural strengths

- L1 Markdown remains the source of truth; SQLite is disposable.
- No daemon, service, remote DB, cloud account, or agent SDK is required.
- Unknown types and broken links are generally treated tolerantly.
- The source skill is only about **128KB**; the generated dist directory is about **84KB**.
- The test seams around injected embeddings are useful and make deterministic tests fast.
- The decision to keep semantic judgment in the host agent is appropriate for ingest/query/lint.

## Findings

### P0: Validator certifies invalid YAML and invalid reserved files

`okflib.parse_frontmatter()` is a line-oriented YAML subset parser. It ignores malformed lines and treats malformed flow values as ordinary strings. `validate_bundle()` therefore cannot prove the SPEC requirement that frontmatter be parseable YAML.

Counterexample executed during this review:

```yaml
---
type: [unterminated
title: "unclosed
---
```

PyYAML rejects it with `ParserError`; Mneme reports:

```text
0 error(s), 0 warning(s)
```

Reserved-file validation is similarly incomplete. Arbitrary frontmatter on a nested `index.md`, malformed log headings, and non-newest-first entries are accepted. `_validate_reserved()` currently checks only whether index/log bodies are empty.

**Impact:** the primary conformance claim is unreliable. This alone blocks 1.0.

**Design decision required:** full YAML conformance and strict zero dependencies conflict. For 1.0 choose explicitly:

- add a small YAML dependency to the validator; or
- define a canonical Mneme producer subset but use a real YAML parser for consuming/validating general OKF bundles.

Do not claim general OKF validation with the current parser.

### P0: Lint calls a nonexistent function

Both SKILL variants and `workflow-lint.md` call `okflib.find_orphans()`. That function does not exist. The documented command raises `AttributeError`.

**Impact:** one of the three foundational LLM Wiki operations cannot run as written.

### P0: Dream is unsafe and mathematically inconsistent

The dream workflow has several independent blockers:

- It interprets `distance <= 0.08` as `similarity >= 0.92`, while `index-design.md` correctly says sqlite-vec returns a raw distance. The real sample search returned distances around `0.93`; the conversion in dream is not valid without a declared metric and normalization formula.
- It depends on the missing `find_orphans()` function.
- It runs `git add -A` before resolving the bundle, so it can commit unrelated user changes or operate on the wrong repository.
- Writing candidates to `.mneme/dream-pending/` and then copying them into place is not a transaction. A partial copy or failed post-validation leaves filesystem changes behind.
- The stated rollback behavior is not implemented by code or a tested protocol.

**Impact:** scheduled dream can modify or commit unintended content. It should be disabled or report-only until it has executable safety tests.

### P0: Delivery model is internally inconsistent

The repository currently mixes two product identities:

1. a self-contained agent skill installed by symlink; and
2. a Python distribution with extras and CLI-like documentation.

Neither delivery path is complete:

- `pip install -e .` installs distribution metadata but exposes no `mneme` module and no `mneme` console command.
- The built wheel contains Python files under `skills/mneme/scripts`, but omits `SKILL.md` and `references/` and defines no entry point.
- `dist/mneme-0.2.0/` contains the skill, but no dependency manifest/build metadata for installing `mneme[index]`.
- SKILL says paths are relative to the skill directory, but most commands use `python3 skills/mneme/scripts/...`. From an installed skill directory, that path is wrong; from an arbitrary project cwd, it usually does not exist.
- The dist directory is manually copied and ignored, with no reproducible build script, manifest, checksum, or parity test.

**Impact:** a developer working in this repository can run the code, but a fresh user cannot reliably install and execute the advertised artifact.

**Decision required:** choose one explicit 1.0 contract.

- **Skill-first:** install the skill directory, resolve its absolute directory at runtime, ship a dependency/bootstrap file, and stop presenting the wheel as the product.
- **CLI + skill:** package a real `mneme` Python module and `[project.scripts]` entry point, include skill resources, and make SKILL call the installed CLI.

The current hybrid is the least elegant option.

### P1: Ingest violates the raw-source and log contracts

The architecture says raw sources are immutable and copied into `sources/`. Current English/Chinese SKILL and `workflow-ingest.md` write concept pages but do not copy the source.

The log contract is newest-first, but ingest instructions say to append the newest entry. Appending makes the log oldest-first over time.

**Impact:** provenance and chronological maintenance degrade during normal use, even if every generated page validates.

### P1: Relative config paths are not persistent

`mneme init wiki` stores:

```toml
bundle_path = "wiki"
```

Resolving that config from a different cwd still returns `wiki`, now pointing at a different location. This was reproduced in two temporary directories.

Paths containing quotes are also written without TOML escaping, and the reader is an ad hoc line parser.

**Impact:** the default documented init form can work once and fail later depending on cwd.

### P1: Long Markdown can be silently truncated in the embedding model

`chunk_markdown()` splits only at Markdown headings. A long page with no headings becomes one chunk. The selected model has a 512-token input limit; FastEmbed tokenization truncates longer input. There is no paragraph/token fallback or overlap despite `index-design.md` claiming paragraph/512-token chunking for sources.

**Impact:** the tail of long concept pages can be absent from semantic search with no warning.

### P1: Retrieval quality has no scientific acceptance test

Current tests use SHA-based fake embeddings. They verify plumbing, not relevance. Missing evaluations include:

- Chinese short-query retrieval with the model's recommended query instruction.
- exact names, IDs, acronyms, and filenames where vector search is weak.
- bilingual queries.
- multiple chunks from one concept dominating top-k.
- no-answer queries and score/gap calibration.
- type-filter recall beyond the bounded over-fetch window.

The BGE model card recommends a Chinese retrieval instruction for short-query-to-passage search, while the current code embeds query and passage identically. The card says no-instruction v1.5 is usable but slightly worse; the correct choice must be measured on Mneme data.

**Impact:** search works on the two-page sample, but its usefulness on the intended 141-document corpus is unknown.

### P1: Model compatibility check is too permissive

When a stored index reports model `custom`, a production query model is accepted. A custom index and a production query may share dimensions while occupying unrelated vector spaces.

**Impact:** a test/fallback index can produce plausible-looking but meaningless production results.

### P1: Dependency stability is below 1.0 standards

`sqlite-vec` describes itself as pre-v1 and explicitly warns to expect breaking changes. Mneme leaves both `sqlite-vec` and `fastembed` unbounded. There is no CI matrix for Python 3.10–3.13 or operating systems.

**Impact:** a clean install can change behavior without a Mneme code change.

### P2: Documentation has multiple sources of truth

- `AGENTS.md` reflects v2.2; `CLAUDE.md` still describes the deleted Strands layer and nonexistent scripts.
- English and Chinese SKILL differ materially. English permits a test fake-embedding fallback after model download failure; Chinese explicitly forbids production use.
- `workflow-query.md` still contains a malformed Markdown citation example.
- Four generations of detailed plans total about 3,880 lines, while runtime Python is about 770 lines. Historical plans are valuable, but they dominate the repository and are easy to mistake for current instructions.

**Impact:** behavior changes depending on which client or document an agent reads.

### P2: Avoidable L2 resource overhead

Measured local costs:

| Item | Cost |
|---|---:|
| Source skill | ~128KB |
| Generated skill dist | ~84KB |
| Development venv | ~374MB |
| BGE model cache | ~91MB |
| Warm search peak memory | ~246MB |
| Reindex peak memory | ~248–350MB, depending on the reported macOS metric |
| 2-concept index | ~2.1MB |
| Warm search latency | ~0.31s |

These costs are not unreasonable for an optional semantic layer, but several are avoidable:

- FastEmbed defaulted to an OS temporary cache on this machine, so the 91MB model may be removed and downloaded again.
- Every reindex embeds every page even when unchanged.
- Reindex calls the embedder and commits metadata/rows concept by concept instead of batching chunks in one transaction.
- Every CLI search starts a new ONNX process and loads the model.
- Top-k is chunk-level and can repeat one concept, wasting agent context.

Do not add a daemon merely to optimize this. First add a stable model cache, batch embedding/transactions, concept-level deduplication, and a cheap lexical path. Measure the real 141-document corpus before implementing incremental indexing.

## Elegance assessment

The project is elegant at the **idea and boundary** level:

- format over platform;
- Markdown source of truth;
- derived local index;
- host agent for judgment;
- optional heavy dependencies.

It is not yet elegant at the **contract and delivery** level. Too much behavior lives in prose that references missing functions, duplicated schema documents drift, and packaging is neither clearly skill-only nor a real CLI package.

The right simplification is not deleting L2. It is reducing the number of truths:

1. one executable workflow API;
2. one canonical schema document with generated client variants;
3. one build command;
4. one conformance implementation;
5. one retrieval benchmark.

## Recommended TDD program

### Phase 0: Freeze dangerous behavior

- Mark dream experimental and report-only.
- Remove the fake-embedding production fallback.
- Make lint fail clearly until orphan analysis exists.
- Keep version below 1.0.

### Phase 1: OKF conformance TDD

Create a table-driven conformance suite derived from SPEC §3–§9.

Required red tests:

- malformed YAML flow/list/quote/indentation;
- YAML scalar types and multiline values;
- empty/non-string `type`;
- unknown keys/types accepted;
- broken links warning-only;
- missing index warning-only;
- nested index frontmatter rejected;
- root index allows only the documented version declaration;
- malformed log dates and non-newest-first order;
- one invalid file does not hide valid concepts.

Acceptance: no known false-positive conformance examples.

### Phase 2: Installation and path TDD

Choose skill-first or CLI+skill before coding.

Required tests from a clean temporary home/cwd:

- install the release artifact only;
- invoke init/reindex/search without repository-relative paths;
- install L2 dependencies from the artifact's documented command;
- initialize with a relative path, switch cwd, and resolve the same absolute bundle;
- paths containing spaces, quotes, and non-ASCII characters;
- build artifact contains exactly the declared files;
- source skill and dist are byte-equivalent after exclusions.

Acceptance: a fresh user can run the quickstart without the source repository.

### Phase 3: Ingest/query/lint workflow TDD

Use a deterministic host-agent fixture or scripted acceptance harness.

Ingest assertions:

- raw source copied once with content hash preserved;
- concept pages conform;
- index links and descriptions updated without duplicates;
- log entry inserted newest-first;
- reindex and validate complete;
- partial failure leaves bundle unchanged or produces an explicit recovery record.

Query assertions:

- reads full pages, not only snippets;
- deduplicates concepts;
- citations resolve;
- reports insufficient coverage on labeled no-answer questions.

Lint assertions:

- orphan analysis exists and is tested;
- invalid YAML is an error;
- broken links remain warnings;
- lint itself performs no writes.

### Phase 4: Retrieval science TDD

Build an evaluation corpus from the first real 141-document ingest.

Minimum query set:

- 20 Chinese semantic paraphrases;
- 10 exact entity/ID/acronym queries;
- 10 cross-document synthesis queries;
- 5 bilingual queries;
- 10 no-answer queries.

Record expected concept IDs and evaluate at least Recall@5, MRR@10, duplicate-concept rate, and no-answer behavior. Compare:

- index/index.md or lexical baseline;
- BGE without query instruction;
- BGE with the recommended Chinese query instruction;
- optional lexical + vector hybrid.

Provisional gates, to be revised after labeling:

- Recall@5 >= 0.85;
- MRR@10 >= 0.70;
- duplicate concept rate in top-10 <= 10%;
- no silent truncation for pages over 512 tokens;
- no-answer questions do not produce confident wiki-backed claims.

### Phase 5: Dream safety TDD

Dream should return only after the lower layers are stable.

Required tests:

- dirty worktree is never auto-committed;
- bundle repository is resolved before git operations;
- no-git mode is report-only;
- similarity math matches the actual sqlite-vec metric;
- every planned mutation is previewed in a manifest;
- validation failure restores the exact pre-dream tree;
- change cap is enforced;
- no push occurs;
- report contains reproducible reasons and commit IDs.

### Phase 6: Resource and release gates

Benchmark the real corpus and set budgets. Suggested initial budgets:

- base skill artifact < 250KB;
- no third-party dependency for reading plain Markdown;
- warm search < 1s on the reference Mac;
- peak search RSS < 350MB with the selected model;
- cached full reindex of the real corpus < 60s;
- stable model cache, no repeated download after reboot/temp cleanup;
- deterministic build with checksum;
- pinned compatible dependency ranges;
- CI on Python 3.10–3.13 and macOS/Linux where sqlite-vec is supported.

## Copy and prompt acceptance corpus

Before 1.0, test not only code but also the skill wording. Use the same temporary bundle and run these user-facing prompts against each supported host:

1. “初始化一个 wiki 到相对路径 `./notes wiki`。”
2. “把这篇笔记摄入知识库，保留原始来源。”
3. “搜索客户画像，只给我页面列表，不要综合回答。”
4. “根据 wiki 回答客户画像和标签体系的关系，带页面引用。”
5. “这个问题 wiki 里没有答案时请明确告诉我。”
6. “检查知识库，但不要修改任何文件。”
7. “找出孤儿页和断链，分别说明严重程度。”
8. “在一个有未提交修改的仓库里运行 dream。”
9. “模型下载失败时继续 ingest。”
10. “从另一个工作目录重新搜索刚才初始化的 wiki。”

Expected behavior should be stored as assertions, not left to subjective review.

## 1.0 release gate

Do not release `1.0.0` merely when the current 41 tests remain green. Release only when:

- all P0 and P1 findings above are resolved or explicitly removed from the product contract;
- install-to-query works from the release artifact in a clean environment;
- the validator passes the conformance counterexample suite;
- ingest/query/lint have end-to-end acceptance tests;
- dream is either safety-tested or excluded from 1.0;
- retrieval meets the labeled real-corpus gates;
- resource budgets and dependency versions are documented and tested;
- AGENTS/CLAUDE/SKILL language variants cannot drift silently;
- the build is reproducible and CI is green.

## Recommended version path

- `0.3.0`: conformance, config/path, missing lint primitives, documentation truth.
- `0.4.0`: clean install/build contract and end-to-end ingest/query/lint harness.
- `0.5.0`: real 141-document dogfood and retrieval benchmark.
- `0.9.0`: release candidate with dream removed or safety-complete.
- `1.0.0`: only after the release gate above passes.

The shortest path to 1.0 is to reduce scope: make **init + ingest + search + query + lint** dependable, keep dream experimental, and avoid converters/MCP until the core contract is proven.
