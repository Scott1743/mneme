# mneme 1.1.0 实施规格

> 上游依据：[1.0 readiness 评估](../reports/2026-07-12-mneme-1.0-readiness-assessment.md) §"P0 Delivery model is internally inconsistent" · [0.3.0 实施规格](2026-07-12-mneme-0.3.0-implementation.md) 模板 · [项目硬约束](../../CLAUDE.md) §"分层依赖" + §"轻量优先"
> 适用基线：commit `5a1c249`（v1.0.0 + skill landing page）
> 目标版本：**1.1.0**（minor bump，向后兼容 CLI API；交付契约从 wheel 切到 skill-first）

## 1. 摘要与边界

本规格把 1.0 readiness 报告 §P0 "Delivery model is internally inconsistent" 列为剩余未关闭项，从混合模型切到 **skill-first 单一交付**：

| 维度 | 1.0（baseline） | 1.1.0（本规格） |
|---|---|---|
| 交付物 | `dist/mneme-1.0.0-py3-none-any.whl` | `skills/mneme/` 整目录 |
| 用户安装路径 | `pip install mneme` | skill.sh → `~/.claude/skills/mneme/` |
| CLI 入口 | `mneme`（setuptools entry point） | `python3 -m mneme ...`（从 skill dir 跑） |
| OKF 核心依赖 | `tomli_w` + 可选 `PyYAML` | **零依赖**（手写 TOML writer + stdlib `tomllib`） |
| L2 依赖 | 装 wheel 时一次性拉 `sqlite-vec + fastembed` | **懒装**：首次 `mneme search/reindex` 才触发 pip install |
| `src/mneme/` 包布局 | 是 | **否**；回归 `skills/mneme/scripts/` 平铺 |
| `pyproject.toml` 用途 | 打包 wheel + 暴露 entry point | **仅 dev/test**；无 `[project.scripts]`、无 `package_data` |
| `dist/` / `mneme.egg-info/` | 存在 | **不存在** |
| CLI API | `mneme init/reindex/search/lint` | **不变** |

**为什么是 minor bump（1.1.0）而不是 major（2.0.0）：**

- CLI 子命令、参数、退出码、JSON 输出 schema **不变** —— pip 装的 1.0.x 用户继续工作
- 唯一破坏性变更 = 交付物格式（不再有 wheel）—— 但这只是新用户的安装入口，老用户无感
- OKF v0.1 协议、bundle 结构、`find_orphans` 算法、L2 索引 schema **不变**
- semver 把"交付物格式变更"归为 MINOR 是允许的（参见 semver §7：当功能以向后兼容的方式增加时）

**为什么不延期到 2.0：**

- 1.0 readiness 报告已明确"P0 交付模型分裂"，是 1.0 release gate 没关上的最后一项
- 现状 skill.sh 用户拿到的是 SKILL.md + 残缺的 `skills/mneme/scripts/mneme/` 目录（v1.0 撤回 src/ 后遗留的 stub），连 `init` 都跑不起来
- 单一交付路径才能让 §CLAUDE.md 硬约束"git-native + 轻量优先 + 分层依赖"真正成立

**范围边界：**

- v1.1.0 **只解决交付模型分裂**。1.0 readiness 报告里的其他 P1（长 Markdown 截断、retrieval benchmark、依赖稳定性）已经分别在 v0.4.0 / v0.5.0 / v0.6.0 阶段关掉或挂测试守住，本规格不再回头
- dream workflow 仍按 v1.0 状态保留（**不在交付模型内**）
- 不引入新的 OKF 协议依赖（v0.1 不变）
- 不变更任何 §CLAUDE.md §"非目标"列出的项（v1 converters / MCP / 多 bundle 管理等仍 deferred）

## 2. 关键设计决策

### 2.1 交付模型 — 采用 A：skill-first，撤回 wheel

**选择 A（推荐）：** 撤回 `src/mneme/` Python 包布局、`pyproject.toml` 的 `[project.scripts]` 与 `package_data`、`dist/` 与 `mneme.egg-info/` 全部删除；SKILL.md 改写为 skill-relative 路径（`python3 skills/mneme/scripts/mneme init ...`）；用户从 skill.sh 安装目录到 `~/.claude/skills/mneme/` 后直接用，OKF 核心零依赖所以无需 bootstrap。

**不选 B（双轨）：** 仍构建 wheel、CHANGELOG 标 deprecated。问题：维护两套路径 = 测试矩阵翻倍（每条命令都要在 wheel install + skill install 下各跑一次），文档两套并行，1.0 已经踩过这个坑（readiness §"P0 Delivery model" 的原话："The current hybrid is the least elegant option"）。

**不选 C（wheel 简化）：** wheel 仍构建但只装 Python 包不带 skill。问题：pip install 出来的 `mneme` 命令能用，但 SKILL.md 仍指向 skill.sh 路径，用户只 pip install 没法跑 scenario（缺 SKILL.md + references）。等于把"两套"换成"半套"。

**理由（三句）：**
1. 1.0 readiness 报告 §"Recommended version path" 早就给出节奏：`1.0.0` 关 release gate → `0.9.0` 是 release candidate → 后续 bugfix 仍走 wheel。本规格直接覆盖到 `1.1.0`，把"路径分裂"这件事一次性闭合。
2. skill.sh 已是本项目公开的唯一分发渠道（commit `5a1c249` 落地了 landing page）；PyPI 从未发布。把 wheel 拿掉 = 文档只需要解释一条路径。
3. `pyproject.toml` 缩成"dev/test 配置单" = 仓库 import 路径、`pip install -e .` 的副作用、构建产物一致性这三类历史问题一次性消失。

**实现要点（不是代码，是约束）：**
- 删除：`src/` 整目录、`mneme.egg-info/`、`dist/`、`pyproject.toml` 的 `[project.scripts]`、`[tool.setuptools.packages.find]`、`[tool.setuptools.package-data]`、`[build-system]` 段
- 回迁：`src/mneme/okflib.py` → `skills/mneme/scripts/okflib.py`，其余同理；`src/mneme/skill/SKILL.md` → `skills/mneme/SKILL.md`，`src/mneme/skill/SKILL cn.md` → `skills/mneme/SKILL cn.md`，`src/mneme/skill/references/*.md` → `skills/mneme/references/*.md`
- `pyproject.toml` 仅保留 `[project]`（name/version/requires-python/optional-dependencies）、`[tool.pytest.*]`、`[tool.coverage.*]`；**新增** `[project.optional-dependencies] index = ["sqlite-vec>=0.1.9,<0.2", "fastembed>=0.8.0,<0.9"]`（L2 懒装目标），`validate = ["PyYAML>=6.0,<7"]`（严格 YAML 校验，可选）
- Makefile 改写：去掉 `python -m build`，新增 release 测试 helper
- `skills/mneme/scripts/mneme` 顶层入口改回 `python3 -m mneme` 风格（`if __name__ == "__main__": sys.exit(main())`）

### 2.2 OKF 核心依赖 — 采用 B：手写 TOML writer + stdlib tomllib

**选择 B（推荐）：** `tomli_w` 替换为 20 行手写 writer，仅服务 `~/.config/mneme/config.toml` 单文件；读仍走 stdlib `tomllib`（3.11+）或 `tomli`（3.10 fallback，通过 `[toml10]` extras）；`PyYAML` 默认不装，仅 `[validate]` extras 可选；其他 OKF 核心 import（re/path/dataclasses/pathlib/argparse/json）全是 stdlib。

**不选 A（保留 tomli_w）：** 把 tomli_w 留作硬依赖也能零感知，但用户装 skill 时必须 `pip install` 一次才能跑 `mneme init` —— 这就破坏了"用户无感知"。tomli_w 是 PyPI 上 `mneme` 唯一的硬依赖（除 transitive），拿掉它 = OKF 核心真正"装上就能跑"。

**不选 C（完全拒绝 TOML）：** 把 `bundle_path` 配置改成 JSON / YAML / SQLite 等。问题：TOML 是 §CLAUDE.md 沿用的 .config 约定（~/.config/mneme/config.toml），改格式等于把"标准工具链能 grep/cat 这个文件"这个能力毁掉。

**理由（三句）：**
1. mneme 写入的 TOML 形态极受限（一个 `[config]` 段下若干 `key = "string"` 行），手写 writer 的转义覆盖度足够（测试 §4.5 守住 quote / backslash / unicode / 新行 / tab 五种 escape 边界）。
2. `tomllib` 是 Python 3.11 stdlib，与 §CLAUDE.md 硬约束"std 优先 + 可选 extras"一致；3.10 fallback 用 `tomli` 走 extras 而不是硬依赖，让 3.10 用户显式 opt-in。
3. PyYAML 是"严格校验 OKF YAML frontmatter"用的（消费端 verify），生产端 parse 走手写子集解析器（与 v0.3.0 一致）。`[validate]` extras 让"想严格"的用户装，"只想用"的用户不装 —— 与 1.0 readiness §"Resource discipline"评分 6/10 的核心建议一致。

**实现要点：**
- 新增 `skills/mneme/scripts/mneme/toml_writer.py`：~30 行；API `write_config(path: Path, data: Mapping) -> None`；仅处理 string / int / float / bool / list[str] / list[int] 五种 scalar + 列表
- `config.py` 的 `import tomli_w` 替换为 `from . import toml_writer as _write`
- §3.2 红测覆盖转义边界（`"` / `\` / `\n` / `\t` / 中文 / 含空格的路径）

### 2.3 L2 依赖 — 采用 C：懒装 + graceful 离线

**选择 C（推荐）：** `mneme search` / `mneme reindex` / `mneme lint`（含 find_orphans 的 L2-aware 路径）首次执行时检测 `sqlite_vec` / `fastembed` import；缺失则触发 `subprocess.check_call([sys.executable, "-m", "pip", "install", "mneme[index]"])`，stderr 显式说明"首次使用语义搜索，正在安装依赖与下载 ~90MB BGE 模型"；成功则继续；失败（offline / permission denied）则 catch `CalledProcessError`，回落到清晰的 SystemExit 提示，告诉用户怎么手动装。

**不选 A（硬依赖 L2）：** OKF 核心用户被迫装 90MB 模型下载 = 80% 用例被惩罚。1.0 readiness §"Resource discipline" 已经把这条列为代价。

**不选 B（纯 opt-in 显式 flag）：** 要求用户先跑 `mneme install --with-index` 再用 search。多一个步骤 = 文档复杂、用户困惑"为什么 init 能跑 search 不能跑"。仍算破坏"零感知"。

**不选 D（彻底删除 L2）：** v0.5.0 Phase 4 dogfood 已经验过 L2 在 142 文档飞书语料上 Recall@5 ≥ 0.85（见 `docs/superpowers/reports/2026-07-12-mneme-0.5.0-bench-results.md`）；v1.0 把它列为 release gate 守门项。删除 = 倒退 6 个月成果。

**理由（三句）：**
1. 90MB 模型下载是唯一无法藏起来的可感知操作（任何方案都得让用户等），但 L2 不是 mneme 的主路径（OKF lint/init/ingest/query 都不依赖它）—— 把它推到"用户主动触发 search/reindex"才出现，是最小可感知化。
2. `pip install mneme[index]` 用当前解释器直接装 = 不需要 venv / 不污染 skill 目录 / 跨平台一致；装在用户已激活的 Python 环境里就跟随用户的 Python 环境（用户管自己的 env）。
3. offline graceful = 不可恢复时给清晰 `SystemExit("install with: pip install 'mneme[index]'")`，不抛 stack trace，与 v0.3.0 §"lint guard"一致的可恢复错误处理风格。

**实现要点：**
- 新增 `skills/mneme/scripts/mneme/lazy_index.py`：~50 行；`ensure_index_deps() -> None` 函数；try import → 失败 → 探测 PEP 668 兼容分支 → subprocess pip install → `os.execvp` 重启 self 让新装包可见 → 失败 → SystemExit
- **PEP 668 分支**（Q3 决议）：`in_venv = sys.prefix != sys.base_prefix`；venv 内走 `[sys.executable, "-m", "pip", "install", "mneme[index]"]`；system Python 走 `[sys.executable, "-m", "pip", "install", "--user", "mneme[index]"]` 绕开 externally-managed（macOS 14+ 与多数 PEP 668 distro）；都不行抛 SystemExit 不抛 stack trace
- **重启机制**：subprocess pip install 成功后当前 Python 进程仍没看到新装包；用 `os.execvp(sys.executable, [sys.executable, "-m", "mneme", *argv])` 重启 self，argv 透传原命令（用户感知 = "首次 search 等了几秒然后自动继续"）
- `cli.py` 的 `cmd_reindex` / `cmd_search` 在 import `indexlib` 之前先调 `ensure_index_deps()`
- `indexlib.py` 不动 lazy 检测逻辑（保持独立可测）
- SKILL.md 显式记录：`mneme search` / `reindex` 首次调用会装依赖 + 下模型；后续调用无副作用；用户可在 SKILL.md 搜 `pip install mneme[index]` 看到 offline fallback 指引（offline 时 SystemExit 提示）

### 2.4 skill.sh 安装路径处理 — 采用 A：SKILL.md 用 skill-relative 路径

**选择 A（推荐）：** SKILL.md 与 references/*.md 全部改写为 skill-relative 路径：`<skill-dir>/scripts/mneme.py init ...`；用 `<skill-dir>` 占位符，agent 通过 `pwd` + `~/.claude/skills/mneme` 解析（约定位置）；CLI 入口走 `python3 <resolved-skill-dir>/scripts/mneme.py` 而非 `mneme`。

**不选 B（依赖 PATH）：** skill.sh 不暴露 post-install hook（v1.0 调研结论：skill.sh 只复制目录到 `~/.claude/skills/<name>/`，不做 PATH 操作），要 PATH 生效得用户手动 `export PATH=...`，破坏"零感知"。

**不选 C（环境变量自发现）：** 让 SKILL.md 用 `${MNEME_HOME}/scripts/mneme.py`。问题：agent 不会主动 export 这个变量；要 export 就得在 SKILL.md frontmatter 或某个 install hook 里说明，复杂度高于 A。

**理由（三句）：**
1. skill.sh 的安装路径是约定位置 `~/.claude/skills/<name>/`，agent 通过 Bash 调 `python3` 时可以拼这条路径，无需运行时自发现。
2. SKILL.md 是给 agent 读的 Markdown，不是给 shell 跑的脚本 —— 完整路径写在 Markdown 里 = grep / 静态分析 / 文档截图都能直接验证。
3. `python3 <path>/scripts/mneme.py` 与 `python3 -m mneme` 等价（顶层 `__main__.py` 已注册），路径变体不影响 CLI 行为。

**实现要点：**
- SKILL.md §"Step 0" 增加："skill 安装位置：`~/.claude/skills/mneme/`（skill.sh 默认）。agent 调用 CLI 用 `python3 ~/.claude/skills/mneme/scripts/mneme.py <subcmd> [args]`"
- references/*.md 中所有 `python3 scripts/mneme.py` → `python3 ~/.claude/skills/mneme/scripts/mneme.py`（或 `<skill-dir>/scripts/mneme.py`）
- §3.4 红测用 grep 守住"无 repo-relative 路径" + "有 skill-relative 路径"
