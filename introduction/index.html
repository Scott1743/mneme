I. Why this exists

我为什么不想再让 LLM "重新检索"了。

RAG 是冷的缓存。你三周前读过的论文里那句话，跟你今天写的那段代码——它不知道。除非那句话被你写进过某个属于你的地方。Dream 把读过的资料编译进一棵本地 Markdown 树；search 在那棵树上走，agent 不用每次从零开始。

这棵树只用三种规则（YAML frontmatter、一个 type 字段、保留 index.md 与 log.md），全部来自 OKF v0.1。普通目录，能 cat、git clone、diff blame branch，能被你写的脚本解析。LLM Wiki 的这个思路最早由 Karpathy 在 2026 年 4 月的 gist 里摊开——Mneme 只是把它做成一个能落到 ~/.claude/skills/mneme/ 的小工具。

II. First prompts

一行命令，然后去跟你的 agent 聊。

重启 agent 会话。需要记住什么时发起 dream，需要找回来时交给 search；每次知识写入仍先由你确认。

dream · 写

把读过的资料编译进树

论文、网页、对话、PDF 里的文字——丢给 agent。agent 调 mneme dream，给你一份预览（哪些页面要新建、哪些要更新、tag 怎么设），你点头之后才落盘。

search · 读

在树里走

你问一个问题，agent 沿 index.md、tags、链接与本地 Graph 走，读完整页 再给答案——而不是把命中的 chunk 拼回来。引用使用以 / 开头的 bundle-relative 路径。

把刚读的这篇 OKF v0.1 编进我的 wiki，
按 Source 摘要成概念页。

mneme dream 永远是 read-only 报告，没有 --apply 这种写入捷径——落不落盘由你决定，不是脚本。

III. The walk

知识是 graph，不是向量近邻——它得被人走过来。

一张页面要先被写出来，再能被找到。Mneme 把"被找到"这件事分给三个不同形态的标记：

type 描述这份文档做什么（Concept、Reference、Source、Summary，或你自己造的新类型）。tags 描述这份文档谈什么（无 schema，由 wiki 自己长出来）。Markdown 链接描述这份文档和哪份文档有关——这是 search 的骨头，agent 沿链接在 graph 里走，而不是按近邻相似度从一堆 embedding 里拽回来。

不能反过来用：type 在 frontmatter 里有协议语义（OKF §4.1）；tags 是你想给它什么语义就是什么——OKF 留给写者的自由。

v4.0 在 FTS5 之前增加一层可删除的本地 Graph：mneme reindex --graph 从现有页面、tags 和 Markdown links 派生 .mneme/graph.db，并刷新 .mneme/fts.db。Graph 不拥有事实；它消失时，wiki 仍可由 agent 直接读取，搜索也回退到 FTS5/L0。Graph 存在且 L2 未激活时，普通 search 自动使用 hybrid；需要诊断时可用 --mode graph|fts|hybrid。语义召回仍通过用户明确执行一次 reindex --l2 单独启用。

v4.1 又补了一层可审核的 Graph enrichment：在获批的 dream 写入后，agent 从完整页面提出实体、关系、证据与置信度，连同页面改动一起给你预览；获批后再由确定性的 mneme graph ingest 写入 Graph。抽取结果可以在重建时重放，也可以和 graph.db 一起删除——它帮助导航，但永远不能盖过 Markdown 原文。

召回的下一步留给另一段路：默认不下载任何外部模型，也不装额外依赖。wiki 本身、agent 的工作流、dream 与 search 的契约都和那段加法无关。

search 没有索引也能跑，
索引烂掉也不会伤到事实。

IV. Why this is what it is

我没有写的那些功能，正是它有用的原因。

没有必须常驻的 GUI，没有服务端进程，没有云同步。装它的那一秒它就在；不装它的那一秒它就不在——这没什么损失。整个运行时是 Python 标准库加一份 ~/.claude/skills/mneme/ 目录下的 Markdown。v4.2 的 mneme serve 也不是例外：它是你显式拉起、Ctrl-C 即停的前台面板（见下节），不是长在后台的服务。

没有矢量模型自动下载，没有权重在后台跑，没有显卡假设，没有你不知道的"隐式索引"在悄悄长。你不会被提醒"已为您下载 X MB 模型"——因为没有人替你下。你不愿意装那种依赖，那就不装；wiki 与 dream / search 仍然完整工作。

Mneme 仍不提供无人监管的事实策展：不会自动改事实正文、碰 raw source、合并归档，也不会 git add、commit 或 push。v4.1 可以在你明确选择后创建每天 02:00 的宿主 agent 健康任务：只报告模式完全不写 wiki；受限自动修复只处理内容已经明确的 frontmatter、tags、内部链接与索引项，并在有歧义、撞上已有改动或涉及超过 5 个概念页时退回报告。CLI mneme dream 本身始终只读。

V. The console

想自己看一眼 wiki 的时候，不必开一场 agent 会话。

v4.2 把同一份知识交到你自己手里：mneme serve --open 启动一个只绑 127.0.0.1 的前台可视化控制台——不注册开机启动、不写 pidfile、不联网，整个服务端只有 Python 标准库；用完 Ctrl-C 即停。六个面板——总览、搜索、浏览、体检、Dream、图谱——把 lint / search / dream / reindex 变成一屏看得见的健康度。

面板是无状态的：它自己没有数据库，每次请求直接读 bundle 与 .mneme/ 缓存。删掉它，wiki 一个字节不少。

P1 是只读 + 可重建的 disposable 缓存：面板里唯一的写按钮是「重建索引」，它只重建 .mneme/ 下的 FTS5 / Graph 缓存，永不写 Markdown 事实。每次启动会打印一个一次性的 session token，配合 Host 头校验与路径沙箱——同机的其他进程和恶意网页都调不动它。

体检页把每条 lint 诊断列成表格，行尾各有一个「复制提示词」按钮：面板不自己动手修，而是把结构化诊断、bundle 路径、规则出处与硬约束组装成一段提示词，你复制给 agent，就进入了正式的修复流程。Dream 页是只读审计视图（OKF 硬规则与写作纪律逐项打勾打叉）；写盘仍走对话里的 dream 审批流——「确认写入」按钮刻意置灰，留给 P2。
