I. Why this exists

把读过的东西编译一遍，让它在每一行回答里继续长出来。

每次提问都重新检索，等于每次都在空仓库里重新发现知识。RAG 是冷的缓存；它不知道你三周前读过的论文里那句话跟你今天写的那段代码相关——除非那句话已经被记进某个属于你的地方。Dream 把读过的资料编译进一棵可以反复行走的本地 Markdown 树；search 在那棵树上走；AI 不是每次都从零开始，而是访问已经在那里的东西。

这棵树只有三种规则（YAML frontmatter、一个 type 字段、保留 index.md 与 log.md），全部出自 OKF v0.1。树根在你手里——一个普通目录，能 cat、能 git clone、能 diff blame branch。能用别人写的工具打开，也能被你自己写的脚本解析。

当代的 LLM Wiki 思路最早由 Andrej Karpathy 在 2026 年 4 月的 gist 里摊开：他问过一个朴素得让人不安的问题——为什么没有人把知识像源码一样维护？ Mneme 不发明这个想法；它只是想把这个想法做成一个真正能落到你 ~/.claude/skills/mneme/ 的小工具。

II. First prompts

落地之后，和你的 Agent 聊。

装上 mneme 技能，重启 agent 会话。从这一秒起，你和你的 agent 之间多了一个默契：你丢任何想法进来，它都会维护你的树。

dream · 写

把读过的资料编译进树

把论文、网页、对话、PDF——任何 markdown 文本——丢给 agent。agent 调 mneme dream，给你一份预览（哪些页面要新建、哪些要更新、tag 怎么设），你点头之后它落盘。

search · 读

在树里走

你问一个问题，agent 先看 index.md，沿着 tags 与链接走，读完整页 再给答案——而不是把命中的 chunk 拼回来。答案里有引用，哪段来自哪页，bundle-relative 路径。

你刚读了一篇关于 OKF v0.1 的文章，
帮我把它编进我的 wiki，按 Source 摘要成概念页。

mneme dream 的输出始终是 read-only 报告，没有任何 --apply 的写入捷径——这个纪律是为了让你，而不是脚本，决定改动是否落盘。

III. The walk

knowledge 是引用，不是索引；它在 graph 里被 walk 到。

一张页面要够好，先要被 agent 写出来；下一步要够有用，它要能被其它页面 找到。Mneme 把这个“被找到”这件事分给三个不同形态的标记：

type 描述这份文档做什么——是 Concept（概念）、Reference（外部资料的蒸馏）、Source（原始资料指针）、还是 Summary（合页）。OKF 不要求 type 注册，所以你能根据自己 wiki 的需要造新的 type。

tags 描述这份文档谈什么——是一个无 schema 的标签数组；没有全局 taxonomy，词汇由这座 wiki 自己长出来。

Markdown 链接描述这份文档和哪份文档有关——具体是谁给谁交代、谁是前提、谁是参考。这是 search 的骨头：agent 按 bundle-relative 链接在 graph 里走，而不是按近邻相似度从一堆 embedding 里拽回来。

vocab 不能反过来用：type 在 frontmatter 的位置上是有协议语义的（OKF §4.1 列在外）；tags 是你想给它什么语义就是什么——这是 OKF 留给写者的自由。

当你的 wiki 小于几百页时，grep 与 Read 已经足够。当它更大，Mneme 会在第一次 reindex 时安静地打开一个本地 SQLite FTS5 索引（零依赖、零网络），wiki 越大索引越有用。索引文件 .mneme/index.db 是可删除的派生缓存——它从来没拥有事实，只是帮 agent 走快两步。

召回的下一步留给另一段路：默认 不下载任何外部模型，也 不装额外的依赖。wiki 本身、Agent 的工作流、dream 与 search 的契约都和有没有那段加法无关。

所以 search 没有索引也能跑，
索引烂掉也不会伤到事实。

IV. Why this is what it is

Mneme 选择留下的，刚好是让你不再需要运维的那些东西。

没有 GUI，没有 web 应用，没有服务端进程，没有云同步。装它的那一秒，它就在那里；不装它的那一秒，它就不在——这没什么损失。整个工具的运行时是 Python 标准库加一份 ~/.claude/skills/mneme/ 目录下的 Markdown。

没有矢量模型自动下载，没有权重在后台跑、没有显卡假设、没有你不知道的"隐式索引"在悄悄长。你不会被提醒"已为您下载 X MB 模型"——因为没有人替你下。你不愿意装那种依赖，那就不装；wiki 与所有 dream / search 仍然完整工作。

定时自动策展（“mneme dream 自动每天跑、自动 commit、自动归档”——v0.1 里曾经存在过的形态）刻意不发货。一个会自己改你的事实页、自动向你 git commit -A 的技能，需要先跨过安全门禁（相似度数学 / find-orphans / git-add-A 时序 / 事务暂存 / rollback 测试 五项 P0），那是另一个项目的事了。在这五项没全被覆盖之前，mneme dream 只输出报告——它把"是否落盘"的决定留给你。

V. Friends of Mneme

同一个方法论，另一个落点。

朋友项目

森林密语 / 塔罗树洞 · tarot-confessional

把一次塔罗解读或长会话里有价值的体会，用同一套“本地优先 / Agent 驱动 / 可检查的开放文件”原则整理进你自己的长期 wiki。两边都坚持：本地优先、写入显式——你不主动「保存到 Wiki」，什么都不会被留下。Mneme 提供整理的工具；森林密语提供抽牌码与象征性解读。
